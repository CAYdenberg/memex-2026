❯ Ok, having done all that, is there any way I can create a new tauri command   
that can interact with the sender that has been created in                      
create_relay_from_endpoint? Don't change anything, just explain it.             
                                                                                
● Yes, but it requires shared state. The fundamental problem is that sender     
  lives inside a spawned background task — no other code can reach it directly. 
  You need a way to put it somewhere both the task and a new command can access.
                                                                                
  The Tauri-idiomatic approach: managed state                                   
                                                                                
  Tauri has a state management system where you register a value with the app at
   startup, and any command can receive it as a parameter via State<T>. The
  pattern would be:                                                             
                  
  1. Define a state struct holding an Option<Sender> wrapped in a Mutex (needed 
  because state must be Send + Sync and commands can run concurrently):
  struct GossipState(Mutex<Option<GossipSender>>);                              
  2. Register it with the app at startup (in lib.rs) with                       
  app.manage(GossipState(Mutex::new(None))).                                    
  3. Pass the AppHandle (or State<GossipState>) into create_relay_from_endpoint.
   After split(), store the sender into the state:                              
  *state.0.lock().unwrap() = Some(sender);
  4. A new command like send_message can then receive State<GossipState>, lock  
  it, check the Option is Some, and call sender.broadcast(...).                 
                                                                                
  The key design considerations:                                                
                  
  - The Option is necessary because the sender doesn't exist until create_share 
  has been called. A send_message command called before that needs to handle the
   None case gracefully (return an error like "not connected").                 
  - If you want multiple concurrent shares (multiple URLs), you'd need a
  HashMap<String, Sender> keyed by URL instead of a single Option.              
  - Mutex here is fine since sending a message is fast and non-async. If you
  needed to .await while holding the lock you'd need tokio::sync::Mutex instead 
  of std::sync::Mutex.
